# AI Vercel Deploy Guide

This document is for AI agents that need to deploy a local or cloned web project to Vercel without a dedicated Vercel MCP/plugin. Treat it as a decision tree: inspect the project first, then choose the least manual path that fits the current environment.

## Goal

Deploy the project and return the final Vercel URL. Prefer actions the agent can perform directly through the Vercel CLI. Ask the user only for credentials or approvals that cannot be created safely by the agent.

## First Inspect

Run these checks from the project root:

```bash
git remote get-url origin 2>/dev/null
cat .vercel/project.json 2>/dev/null || cat .vercel/repo.json 2>/dev/null
test -f package.json && cat package.json
test -f vercel.json && cat vercel.json
test -f .env.example && cat .env.example
```

Use the results to identify:

- Whether the project is already linked to Vercel.
- Whether the project has a GitHub remote.
- The framework and build command.
- Required environment variables.
- Whether the app uses Vercel Blob, Postgres, KV, or other managed resources.

## Authentication Decision Tree

Use this order:

1. If `VERCEL_TOKEN` exists in the environment, use it. Do not print it and do not pass it with `--token`.
2. If no token exists, run `vercel whoami`.
3. If the CLI is logged in, proceed with the existing login.
4. If the CLI is not logged in, open the login page and run `vercel login`:
   ```bash
   open https://vercel.com/login
   vercel login
   ```
5. If the user wants repeatable unattended deploys, open the token page and ask them to create and provide a Vercel token:
   ```bash
   open https://vercel.com/account/tokens
   ```

Do not make the user find `VERCEL_ORG_ID` or `VERCEL_PROJECT_ID` when `.vercel/project.json` already exists. Read them from the file or let the CLI use the link.

## Project Linking

If `.vercel/project.json` or `.vercel/repo.json` exists, treat the project as linked.

If no link exists:

- If there is a git remote, prefer:
  ```bash
  vercel link --repo -y
  ```
- If there is no git remote, use:
  ```bash
  vercel link -y
  ```

Only ask for team/project choice when the CLI prompts or the user has multiple teams and no local link.

## Environment Variables

Read `.env.example`, docs, and source code to identify required variables. Then prefer CLI setup:

```bash
vercel env ls
printf "%s" "$VALUE" | vercel env add NAME production
printf "%s" "$VALUE" | vercel env add NAME preview
printf "%s" "$VALUE" | vercel env add NAME development
vercel env pull .env.local
```

For secrets generated by the agent, such as `ADMIN_PASSWORD` or `SESSION_SECRET`, generate a strong value and show it to the user once if they need to log in later.

Do not ask the user to manually add env vars in the dashboard before trying the CLI.

## Vercel Blob

If the project imports `@vercel/blob` or references Blob variables, handle Blob before deploy.

First inspect:

```bash
rg -n "@vercel/blob|BLOB_|VERCEL_OIDC_TOKEN|BLOB_STORE_ID" .
vercel blob list-stores
```

If no connected store exists, try CLI creation:

```bash
vercel blob create-store <project-name>-blob --access public
vercel env pull .env.local
```

Notes:

- Connected Blob stores can use `BLOB_STORE_ID` plus runtime `VERCEL_OIDC_TOKEN`.
- Some apps explicitly check `BLOB_READ_WRITE_TOKEN`; if the code requires it and Vercel did not add it, ask only for that missing token or guide the user to the Blob store's env page.
- Do not tell the user to manually create Blob until the CLI path fails.

## Deploy

Default to preview unless the user explicitly asks for production/live:

```bash
vercel deploy -y --no-wait
```

For production:

```bash
vercel deploy --prod -y --no-wait
```

After deployment:

```bash
vercel inspect <deployment-url>
```

Return the deployment URL and summarize any setup performed.

## Git Push Alternative

If the project is linked to Vercel through Git and the user explicitly asks to push or publish through GitHub:

```bash
git status --short
git add <intended-files>
git commit -m "<message>"
git push
```

Do not push without explicit user approval.

## Failure Handling

- If CLI auth fails, open login/token setup and explain the smallest credential needed.
- If `vercel blob create-store` fails, report the error and then fall back to dashboard guidance.
- If env add/update fails because a variable exists, use `vercel env update`.
- If deployment fails, use `vercel inspect <deployment-url> --logs`.
- If the build needs missing secrets, add them through `vercel env` and redeploy.

## Safety Rules

- Never print tokens or long-lived secrets unless the agent generated a one-time password for the user.
- Never put tokens in command arguments where shell history can capture them.
- Do not edit `.vercel/` files manually.
- Do not delete Blob stores, projects, deployments, or env vars unless the user explicitly asks.
- Prefer CLI automation, then browser-open guidance, then dashboard instructions as the final fallback.
